# TaskInsight — 업무 규칙 문서

> **P0 필수 문서.** 이 문서에 없는 규칙은 구현하지 않습니다.
> 모호한 케이스는 이 문서를 먼저 확인하고, 없으면 반드시 확인 요청합니다.

---

## 1. 이슈 번호 체계

| 규칙 | 내용 |
|---|---|
| 번호 방식 | 전역 순번. `issues.id` (SERIAL). 별도 프로젝트별 번호 없음. |
| 표시 방식 | `#123` 형태. 항상 `#` 접두사 포함. |
| 마이그레이션 | Redmine 원본 번호는 `redmine_id` 컬럼에 보존. 새 ID로 재채번. |
| 참조 표기 | 코멘트/설명에서 `#123` 언급 시 자동 링크로 변환. |

---

## 2. 이슈 상태 전환 규칙

### 2.1 기본 상태 목록

| id | 이름 | flow_stage | is_closed |
|---|---|---|---|
| 1 | 신규 | backlog | false |
| 2 | 진행 중 | in_progress | false |
| 3 | 검수 요청 | review | false |
| 4 | 검수 중 | review | false |
| 5 | 완료 | done | true |
| 6 | 반려 | rejected | true |
| 7 | 보류 | blocked | false |
| 8 | 재작업 | rework | false |

### 2.2 전환 허용 규칙

```
신규(1)        → 진행 중(2), 보류(7)
진행 중(2)     → 검수 요청(3), 보류(7), 신규(1)
검수 요청(3)   → 검수 중(4), 재작업(8), 반려(6)
검수 중(4)     → 완료(5), 재작업(8), 반려(6)
재작업(8)      → 진행 중(2), 보류(7)
보류(7)        → 신규(1), 진행 중(2)
완료(5)        → 재작업(8)   ← 재오픈 가능 (project_manager 이상만)
반려(6)        → 신규(1)     ← 재오픈 가능 (project_manager 이상만)
```

### 2.3 상태 전환 시 자동 동작

| 전환 | 자동 동작 |
|---|---|
| → 완료(5) | `closed_at = NOW()`, `done_ratio = 100` |
| → 반려(6) | `closed_at = NOW()` |
| 완료/반려 → 다른 상태 (재오픈) | `closed_at = NULL` |
| → 진행 중(2) | 담당자 없으면 경고 (차단하지 않음) |
| 담당자 변경 | 새 담당자에게 알림 발송 |

### 2.4 이슈 삭제 규칙

- **하드 삭제 없음.** 이슈는 삭제하지 않고 반려(6) 처리만 가능.
- 예외: `system_admin`이 명시적 "영구 삭제" 요청 시에만 하드 삭제 (별도 확인 모달 필요).
- 하위 이슈가 있으면 부모 이슈 삭제 불가 → 먼저 하위 이슈 처리 필요.

## 3. 이슈 계층 (부모-자식)

| 규칙 | 내용 |
|---|---|
| 깊이 제한 | 최대 2단계 (부모 → 자식). 자식의 자식 불가. |
| 순환 참조 | `parent_id = self.id` 또는 순환 구조 저장 시 400 오류 |
| 다른 프로젝트 | 부모와 자식은 반드시 같은 프로젝트 |
| 부모 완료 조건 | 자식이 모두 완료되지 않아도 부모 완료 가능 (경고만 표시) |
| 진행률 계산 | 부모의 `done_ratio`는 수동 입력. 자식 평균으로 자동 계산 안 함. |

---

## 4. 담당자 배정 규칙

| 규칙 | 내용 |
|---|---|
| 담당자 자격 | 해당 프로젝트 멤버여야 함 (`project_members`에 존재) |
| 미할당 허용 | `assignee_id = NULL` 허용. 이슈 목록에서 "미할당"으로 표시. |
| 비활성 사용자 | 비활성 사용자로 담당자 변경 불가. 기존 담당자가 비활성화되면 그대로 유지하되 경고 표시. |
| 자동 배정 | 프로젝트에 `default_assignee_id` 설정 시 신규 이슈 생성 시 자동 배정. 생성자가 명시적으로 지정하면 override. |

## 6. 진행률 (done_ratio)

| 규칙 | 내용 |
|---|---|
| 입력 범위 | 0 ~ 100 (정수). 소수점 불가. |
| 기본값 | 신규 이슈 생성 시 0. |
| 완료 상태 | 상태가 완료(5)로 변경되면 자동으로 100 설정. |
| 수동 override | 완료 상태에서도 수동으로 done_ratio 변경 불가 (완료=100 고정). |

---

## 7. 기한 (due_date)

| 규칙 | 내용 |
|---|---|
| 필수 여부 | 선택. 없어도 이슈 생성 가능. |
| 과거 기한 | 과거 날짜 설정 허용 (경고만, 차단 안 함). |
| 기한 D-1 알림 | 매일 오전 9시 배치: 기한이 내일인 미완료 이슈의 담당자에게 알림 발송. |
| 기한 초과 표시 | UI에서 기한 지난 이슈는 날짜를 빨간색으로 표시. |
| 자식 이슈 기한 | 부모 기한보다 늦은 자식 기한 허용 (경고만). |

## 8. 비공개 이슈 (is_private)

| 규칙 | 내용 |
|---|---|
| 기본값 | false (공개) |
| 비공개 이슈 조회 | 등록자, 담당자, project_manager, system_admin만 조회 가능. |
| 비공개 설정 권한 | 등록자 또는 project_manager 이상. |
| 이슈 목록 | viewer에게는 비공개 이슈가 아예 보이지 않음. |

## 9. 코멘트 (Journal Notes)

| 규칙 | 내용 |
|---|---|
| 최소 길이 | 1자 이상 (공백만 불가) |
| 최대 길이 | 10,000자 |
| 수정 | 본인 코멘트만 수정 가능. 수정 이력 미표시 (단순 수정). |
| 삭제 | 본인 + project_manager 이상 삭제 가능. |
| @멘션 | `@display_name` 패턴 감지 → 해당 사용자 알림 발송. |
| 코멘트 + 상태변경 | 동시에 가능 (이슈 수정 + notes 필드 함께 전송). |

## 10. 파일 첨부

| 규칙 | 내용 |
|---|---|
| 최대 파일 크기 | 20MB (파일 1개 기준) |
| 허용 확장자 | `.jpg .jpeg .png .gif .webp .pdf .doc .docx .xls .xlsx .ppt .pptx .zip .txt .md .csv .json .xml .hwp` |
| 차단 확장자 | `.exe .bat .sh .cmd .scr .vbs .dll .so` (실행 파일류 전체 차단) |
| 저장 경로 | `UPLOAD_DIR/{issue_id}/{uuid}_{원본파일명}` |
| 이슈당 최대 | 50개 (초과 시 400 오류) |
| 삭제 정책 | 이슈 삭제(비활성화) 시 파일은 보존. 수동 삭제만 가능. |
| 다운로드 권한 | 해당 이슈 조회 가능한 사용자 전체 (viewer 포함) |

## 11. 타임 트래킹 (time_entries)

| 규칙 | 내용 |
|---|---|
| 최소 입력 | 0.25시간 (15분) 단위 |
| 최대 입력 | 24.0시간 (하루 기준) |
| 날짜 | 미래 날짜 입력 불가 |
| 수정/삭제 | 본인 기록만 수정/삭제 가능. project_manager는 전체. |
| 이슈 없는 기록 | project_id만 있고 issue_id = NULL인 기록 허용 (프로젝트 공수) |
| 집계 | 이슈별 합산 + 프로젝트별 합산 + 사용자별 합산 모두 지원 |

## 12. 알림 발송 규칙

### 12.1 이벤트별 수신자

| 이벤트 | 수신자 | Teams | 이메일 |
|---|---|---|---|
| 이슈 담당자 배정 (신규) | 새 담당자 | ✅ | ✅ |
| 이슈 담당자 변경 | 새 담당자, 이전 담당자 | ✅ | ✅ |
| 이슈 상태 변경 | 담당자 + 관심 등록자 | ✅ | ✅ |
| 코멘트 추가 | 담당자 + 관심 등록자 | ✅ | ✅ |
| @멘션 | 멘션된 사용자 | ✅ | ✅ |
| 기한 D-1 | 담당자 | ✅ | ✅ |
| 이슈 완료 | 등록자 + 관심 등록자 | ✅ | ❌ |

### 12.2 알림 제외 규칙

- **본인이 발생시킨 이벤트는 본인에게 알림 발송 안 함.**
- 비활성 사용자에게는 알림 발송 안 함.
- 알림 발송 실패 시 로그만 기록 (이슈 저장은 성공으로 처리, 알림 실패가 주요 동작을 막지 않음).

### 12.3 알림 빈도 제한 (Rate Limit)

- 동일 이슈에서 동일 사용자에게 1분 내 같은 타입 알림 중복 발송 안 함.

## 13. 사용자 계정 규칙

| 규칙 | 내용 |
|---|---|
| 이메일 변경 | 불가 (시스템 식별자) |
| 비밀번호 | 최소 8자, 영문+숫자 조합 필수 |
| 로그인 실패 | 5회 연속 실패 시 15분 잠금 |
| 계정 비활성화 | 하드 삭제 없음. 비활성화만. 기존 데이터 유지. |
| 마이그레이션 계정 | 임시 비밀번호 `TempPassword123!`. 최초 로그인 시 변경 강제 (is_password_temp 플래그). |
| 비밀번호 재설정 | 관리자가 임시 비밀번호 설정 가능 (이메일 발송). 셀프 재설정은 v2. |

## 14. 프로젝트 규칙

| 규칙 | 내용 |
|---|---|
| 생성 권한 | system_admin만 |
| identifier | 소문자 영문+숫자+하이픈. 한글 불가. 변경 불가. 최대 100자. |
| 비활성화 | 이슈 있어도 비활성화 가능. 비활성 프로젝트 = 신규 이슈 생성 불가. |
| 삭제 | 없음. 비활성화만. |
| 멤버 제거 | 해당 멤버가 담당자인 이슈는 미할당으로 전환. |

## 15. 검색/필터 규칙

| 규칙 | 내용 |
|---|---|
| 기본 정렬 | `updated_at DESC` |
| 페이지네이션 | offset 기반. 기본 50건, 최대 200건. |
| 검색 대상 | `subject` (제목) + `description` (설명). 저널 내용은 미포함. |
| 검색 방식 | PostgreSQL `ILIKE '%keyword%'` (대소문자 무시). 전문 검색(FTS)은 v2. |
| 필터 조합 | `AND` 조건. 복수 필터는 모두 만족하는 이슈만 반환. |
| 삭제 이슈 표시 | 기본 숨김. `include_closed=true` 파라미터로 완료/반려 포함. |

## 16. 분석 데이터 연동 규칙 (Phase 4)

Phase 1~3에서는 기존 `raw_redmine_*` → `fct_*` ETL 그대로 유지.

Phase 4 전환 시:
- `issues` 테이블 → `fct_issue_snapshot` ETL 연동
- 이슈 저장 시 `BackgroundTasks`로 즉시 ETL 부분 갱신 트리거
- `raw_redmine_*` 데이터는 아카이브 (Redmine sync 중단)
- LLM 타임라인 설명 캐시(`fct_issue_explanation`)는 `issues.id` 기반으로 전환
