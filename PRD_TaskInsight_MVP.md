# PRD: TaskInsight MVP

**작성일:** 2026-05-21  
**버전:** v1.0  
**상태:** Ready for Development

---

## Problem Statement

팀장과 PM은 Redmine에서 업무 현황을 파악하려 해도 "왜 이 이슈가 오래 걸리는지", "어디서 막히는지", "이번 주 보고에 뭘 넣어야 하는지"를 한 눈에 알 수 없다. Redmine은 데이터 입력 도구이지 분석 도구가 아니기 때문이다.

구체적으로:
- 이슈 하나가 수백 일째 멈춰 있어도 **왜** 멈췄는지 Redmine은 알려주지 않는다
- 매주 팀장이 주간보고를 위해 Redmine을 직접 뒤져 수동으로 정리해야 한다
- "검수 단계에 53건이 몰려 있다"는 사실은 알아도 **어떤 조치**가 필요한지 모른다

---

## Solution

Redmine 데이터를 자동 수집·분석하여 팀장/PM에게 **"어디서 막히는가"**, **"왜 막혔는가"**, **"이번 주 보고할 것 3가지"** 를 즉시 제공하는 한국어 SEI(Software Engineering Intelligence) 대시보드.

핵심 차별점:
- **이슈 타임라인 자동 설명**: 이슈를 클릭하면 LLM이 한국어 보고서체로 "왜 오래 걸렸는지"를 자동 설명
- **주간보고 1-click 생성**: 완료량/정체 현황/완료 예측 3항목을 버튼 하나로 자동 생성
- **Read-only 분석**: Redmine 데이터를 읽기만 하므로 기존 워크플로우 무간섭

## User Stories

### Flow 화면 (흐름 진단)

1. 팀장으로서, 업무 단계별(대기중/진행중/검수중/완료) 이슈 건수와 평균 체류일을 펀넬 형태로 보고 싶다. 그래야 어느 단계에서 업무가 쌓이는지 파악할 수 있다.
2. 팀장으로서, 펀넬의 특정 단계를 클릭하면 해당 단계의 이슈 목록이 체류일 내림차순으로 보이길 원한다. 그래야 가장 오래된 이슈부터 살펴볼 수 있다.
3. 팀장으로서, 이슈 목록에서 이슈번호/제목/담당자/체류일/위험점수를 한 눈에 보고 싶다. 그래야 우선 확인할 이슈를 빠르게 선별할 수 있다.
4. 팀장으로서, 이슈를 클릭하면 해당 이슈의 상태 전환 타임라인이 모달로 열리길 원한다. 그래야 이슈의 역사를 시각적으로 파악할 수 있다.
5. 팀장으로서, 이슈 타임라인 모달에서 LLM이 생성한 한국어 설명("왜 오래 걸렸는지")을 보고 싶다. 그래야 데이터를 직접 해석하지 않아도 된다.
6. 팀장으로서, 이슈 타임라인 모달에서 "Redmine에서 열기" 링크를 클릭해 해당 이슈로 바로 이동하고 싶다. 그래야 조치를 빠르게 취할 수 있다.
7. 팀장으로서, 글로벌 필터에서 프로젝트와 기간(1/3/6/12개월)을 선택하면 Flow 화면 전체가 필터링되길 원한다. 그래야 특정 프로젝트만 집중해서 볼 수 있다.
8. 팀장으로서, 이슈 타임라인 설명이 이미 생성된 경우 즉시 보이길 원한다. 처음 요청하는 경우만 대기 시간이 있어도 된다.
9. PM으로서, 위험점수가 높은 이슈(지연, 장기 보류)를 시각적 강조(🔴)와 함께 쉽게 식별하고 싶다. 그래야 우선순위 파악이 쉽다.

### 주간보고 화면

10. 팀장으로서, "이번 주 보고서 생성" 버튼을 누르면 주간보고가 자동 생성되길 원한다. 그래야 회의 전 수동 정리 시간을 아낄 수 있다.
11. 팀장으로서, 주간보고에 ① 이번 주 완료량(전주 대비) ② 정체 구간 현황 ③ 완료 예측 변화 3항목이 포함되길 원한다. 이 3가지가 주간 회의의 핵심 아젠다이기 때문이다.
12. 팀장으로서, 각 항목에 LLM이 생성한 한 줄 해석이 함께 표시되길 원한다. 그래야 숫자의 의미를 별도로 해석할 필요가 없다.
13. 팀장으로서, 한 번 생성된 보고서가 저장되어 다음에 열어도 그대로 보이길 원한다. 그래야 회의 중에도 안정적으로 참조할 수 있다.
14. 팀장으로서, 이전 주 보고서도 조회할 수 있길 원한다. 그래야 추이 변화를 파악할 수 있다.
15. 팀장으로서, 보고서 화면이 브라우저 인쇄(Ctrl+P)에 최적화되어 있길 원한다. 그래야 필요 시 출력해서 활용할 수 있다.
16. PM으로서, 글로벌 필터의 프로젝트/기간 설정이 주간보고에도 동일하게 적용되길 원한다. 그래야 특정 프로젝트만 보고할 수 있다.

### Dashboard 화면

17. 팀장으로서, 전체 소요일/주간 완료량/완료 예측을 한 눈에 보는 Speed 섹션을 원한다. 그래야 팀 속도를 즉시 파악할 수 있다.
18. 팀장으로서, 업무 쏠림 지수/진행 중 업무 건수/미할당 건수를 보는 Effectiveness 섹션을 원한다. 그래야 업무 불균형을 발견할 수 있다.
19. 팀장으로서, 재작업 비율/반려율/이상 신호 건수를 보는 Quality 섹션을 원한다. 그래야 품질 이슈를 조기에 감지할 수 있다.
20. 팀장으로서, 각 KPI 카드에 숫자 + 전기간 대비 추이 + 한 줄 해석이 함께 표시되길 원한다. 그래야 숫자의 의미를 즉시 이해할 수 있다.
21. 팀장으로서, 글로벌 필터가 Dashboard에도 동일하게 적용되길 원한다.

### Settings 화면 (Redmine 연동)

22. 관리자(본인)로서, Redmine URL과 API 키를 Settings 화면에서 입력하고 저장할 수 있길 원한다.
23. 관리자로서, "연결 테스트" 버튼을 눌러 Redmine과 정상 연결되는지 즉시 확인하고 싶다.
24. 관리자로서, 마지막 동기화 시각과 상태(정상/오류)를 Settings 화면에서 확인하고 싶다.
25. 관리자로서, "지금 동기화" 버튼으로 Redmine 데이터를 즉시 수동 갱신할 수 있길 원한다.
26. 관리자로서, Settings UI가 향후 Jira, Asana 등 다른 도구 연동도 추가할 수 있는 구조임을 인지할 수 있길 원한다 (coming_soon 표시).

### 데이터 파이프라인 (시스템)

27. 시스템으로서, Redmine sync 완료 후 ETL이 자동으로 트리거되어 분석 데이터가 항상 최신 상태를 유지해야 한다.
28. 시스템으로서, 매일 새벽 2시에 위험점수 상위 20건의 LLM 타임라인 설명을 사전 생성해 저장해야 한다. 그래야 팀장이 아침에 이슈를 클릭할 때 즉시 볼 수 있다.
29. 시스템으로서, 이슈에 새 저널이 도착하면 해당 이슈의 캐시를 stale 처리하여 다음 요청 시 재생성해야 한다.

## Implementation Decisions

### 아키텍처 원칙
- **Read-only**: Redmine 데이터를 읽기만 함. 쓰기 API 호출 없음.
- **로컬 우선**: 모든 처리는 Mac Studio 로컬에서. 외부 API 비용 없음.
- **Connector 추상화**: Settings의 Redmine 연동은 BaseConnector ABC 위에 구현. 향후 Jira 등 추가 가능한 구조.

### 데이터 파이프라인
- Redmine sync(1시간 주기) 완료 후 ETL 자동 연계
- raw_* 테이블명을 raw_redmine_* 로 rename (alembic 마이그레이션, 데이터 무손실)
- ETL → mart 테이블(fct_*) 갱신 → 분석 API 제공

### LLM 전략
- **이슈 타임라인 설명 모델**: `qwen3.6:35b-a3b` (한국어 품질 우선)
- **관찰 레이어(Narrator)**: `qwen2.5-coder:14b` (구조화 출력)
- **LLM 입력**: 상태 전환 이력 + 담당자 변경 저널 + 코멘트 있는 저널만 선별 (노이즈 최소화)
- **캐싱**: 위험점수 상위 20건 매일 새벽 2시 사전 생성 + 온디맨드 캐시. 새 저널 도착 시 stale 처리.
- **시스템 프롬프트 원칙**: 한국어 보고서체(~함, ~됨). "약점/문제점/부진" 금지. 상황 설명 + 권장 조치 1~2가지.

### DB 스키마 변경
- `raw_*` → `raw_redmine_*` rename (6개 테이블)
- `connector_instance` 신규: Redmine 연결 설정 저장
- `fct_issue_explanation` 신규: LLM 설명 캐시 (issue_id PK, explanation, generated_at, is_stale)
- `fct_weekly_report` 신규: 주간보고 저장 (throughput, bottleneck_summary JSONB, forecast, narrative_text)

### API 계약
| Endpoint | 용도 |
|---|---|
| GET /v1/flow/stages | 단계별 건수 + 평균 체류일 |
| GET /v1/flow/issues | 이슈 목록 (단계/프로젝트/기간 필터) |
| GET /v1/flow/issue/{id}/explanation | LLM 설명 (캐시 우선, 없으면 온디맨드 생성) |
| POST /v1/reports/weekly/generate | 주간보고 생성 (수동 트리거) |
| GET /v1/reports/weekly/latest | 최근 보고서 조회 |
| GET /v1/dashboard/summary | Speed + Effectiveness + Quality 지표 |
| GET /v1/connectors/instances | 연동 목록 |
| POST /v1/connectors/test | 연결 테스트 (저장 전) |
| PUT /v1/connectors/{id} | 연동 설정 수정 |

기존 9개 엔드포인트(`/v1/executive/*`, `/v1/manager/*` 등) 전부 유지.

### 글로벌 필터
- 프로젝트 선택 + 기간 (1/3/6/12개월) 2개만 MVP에서 제공
- 팀원 필터는 v2
- 필터 상태는 모든 화면(Flow/Dashboard/주간보고)에 공유 적용

### UI 원칙
- 사이드바 72px, 5개 메뉴 (Flow / 주간보고 / Dashboard / Settings + 기존 메뉴)
- KPI 카드: 숫자 + 추이 화살표 + 한 줄 해석 3단 구조
- 색상 의미: 빨강(즉시 조치) / 주황(주시) / 회색(정상) / 초록(개선됨)
- 폰트: Pretendard Variable

### 윤리 가드레일
- 개인 비교 화면 없음 (코드 레벨에서 부재)
- 개인에게 단일 점수 부여 금지
- 절대 임계치 없음, 상대 기준만

## Testing Decisions

### 좋은 테스트의 기준
- 내부 구현이 아닌 **외부 동작**만 테스트
- "이 함수가 이 입력을 받으면 이 결과를 반환한다"를 검증
- DB 의존 테스트는 실제 테스트 DB 사용 (mock 금지 — 쿼리 오류 놓침)

### 테스트 대상 모듈
| 모듈 | 테스트 유형 | 이유 |
|---|---|---|
| `issue_explainer.py` | Integration (LLM mock) | 핵심 차별점, 입력→출력 형식 검증 |
| `weekly_report.py` | Unit + Integration | 3항목 계산 로직 정확성 |
| Flow API (`/v1/flow/*`) | API Integration | 필터 조합별 응답 형식 |
| LLM 캐시 로직 | Unit | stale 처리, 캐시 히트/미스 |
| BaseConnector | Unit | 추상화 인터페이스 계약 |
| Redmine sync → ETL 연계 | Integration | 파이프라인 전체 흐름 |

### 테스트 우선순위
1. LLM 설명 캐시 로직 (is_stale 처리)
2. 주간보고 3항목 계산 (throughput delta, bottleneck top3, forecast)
3. Flow API 필터 동작

## Out of Scope

- **Home 화면** ("오늘 살펴볼 것") — v2. MVP에서 `/flow` redirect.
- **Meeting 화면** (면담 준비 자료) — v2. dim_user 확장 필요.
- **팀즈 알림** — v2. Incoming Webhook.
- **PDF 내보내기** — v2. 한국어 폰트 이슈, MVP 복잡도.
- **인증/로그인** — v2. 현재 localhost 단독 사용.
- **Dashboard Business Impact 섹션** — v2. 이슈 분류 태깅 전략 별도 결정 필요.
- **다른 도구 연동 실구현** (Jira, Asana 등) — v2+. 현재 coming_soon 표시만.
- **Redmine 데이터 쓰기** (담당자 변경, 코멘트 등) — 범위 외. Read-only 원칙.
- **개인별 비교 화면** — 영구 제외 (윤리 가드레일).
- **팀원 글로벌 필터** — v2.
- **자연어 질의 인터페이스** — v2.
- **월간보고** — v2.
