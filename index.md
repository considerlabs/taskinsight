1. 핵심 인사이트
브라이언님, 스크린샷만 있으면 “그럴듯한 데모”는 만들 수 있지만, 50명이 문제없이 쓰는 시스템은 “문서 + 데이터 + 테스트 시나리오”가 있어야 가능합니다.
한마디로, AI에게 필요한 건 감이 아니라 판단 기준입니다. 바이브 코딩도 결국 “바이브 운영”까지 가려면 계약서급 명세가 필요합니다.

2. 근거 및 데이터
제가 앞서 캡처에서 파악한 기준으로 이 시스템은 단순 CRUD가 아닙니다.

견적	부가세 포함/별도 여부

입금	초과 입금 허용 여부
프로젝트	완료 조건
리드	고객 전환 조건
리드	중복 고객 판단 기준
일정	반복 일정 지원 여부

로그	어떤 액션을 기록할지
예시
Copy견적 합계 계산:
- 공급가액 = 수량 × 단가
- 부가세 = 공급가액 × 10%
- 총액 = 공급가액 + 부가세

입금 상태:
- 입금합계 = 0이면 미입금
- 0 < 입금합계 < 매출총액이면 부분입금
- 입금합계 = 매출총액이면 입금완료
- 입금합계 > 매출총액이면 초과입금
이 문서 없으면 금액 관련 기능은 만들면 안 됩니다.
진짜입니다. 돈은 농담을 싫어합니다.

10. 대안적 해석 가능성
확인이 필요한 부분도 있습니다.

50명이 전체 사용자 50명인지, 동시접속 50명인지 확인 필요합니다.

전체 50명과 동시 50명은 인프라 설계가 다릅니다.
내부 업무용인지 외부 고객도 접속하는지 확인 필요합니다.

외부 고객 접속이 있으면 보안/권한 설계가 더 강해져야 합니다.
Gmail, Slack, Google Drive가 실제 운영 필수인지 확인 필요합니다.

필수라면 OAuth, 토큰 갱신, 실패 처리까지 설계해야 합니다.
매출/입금이 회계 기준 데이터인지 단순 관리용인지 확인 필요합니다.

회계 기준이면 수정 이력, 잠금, 감사 로그가 필요합니다.
